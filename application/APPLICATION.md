## 為什麼做這個專案 — 新竹教育現場的真實壓力

我住在新竹，這裡是台灣科學園區的核心。園區聚集了超過 15 萬名科技從業人員，工程師家庭密度極高，家長普遍高學歷、重視子女教育。從幼兒園開始讓孩子接觸英語，在新竹不是「提早」，而是「標配」。

**競爭從小一就開始了。** 女兒班上 28 個同學，超過 20 個在上英語補習班。新竹的英語補習班月費約 NT$5,000–15,000，一年下來就是 NT$60,000–180,000（約 US$1,900–5,700）。而且這還只是英語 — 多數孩子同時補數學、鋼琴、程式。家長之間的焦慮是真實的：「別人家的孩子都在學，你的孩子能不學嗎？」

台灣正在推動 2030 雙語國家政策，新竹作為科技重鎮更積極響應 — 雙語實驗小學的入學競爭激烈，家長從幼兒園中班就開始準備。這不是教育焦慮，而是現實：英語能力直接影響孩子未來能否進入理想的學校與職涯。

**但光靠補習班是不夠的。** 我已經送女兒去全美語補習班了，每月上萬元，但回家後她跟我說：「爸爸，老師講的我聽不懂。」全美語環境聽起來很好，但對一個小一的孩子來說，當她跟不上進度、又不敢開口問，補習班的效果就大打折扣。她需要的不是更多課堂時間，而是一個有耐心的、能隨時用中文解釋英文的 1 對 1 助教。

**我是工程師，我能用技術補上這個缺口。** 補習班老師面對 15–20 個孩子，無法在每個孩子開口時都即時糾正、翻譯、回饋 — 但 AI 可以。這個專案不是要取代補習班，而是**輔助**：回家後，女兒可以用自己的節奏、跟 AI 互動複習，說中文就能聽到英文翻譯，不懂的地方可以反覆練習，沒有同儕壓力、沒有趕進度。

這就是一個新竹工程師爸爸的教育熱情：用 AI 輔助補習班，用技術填補課堂學不到的那一塊。

## 專案摘要（300 字內）

我是一個軟體工程師，也是一個小一女兒的爸爸。我開發了兩個完整的 AI 英語教育工具：**chinese2English**（中翻英即時語音翻譯）和 **vocabulary**（12,200 詞彙 + 32 款互動遊戲的英語學習平台）。

chinese2English 已是生產就緒的系統，支援 GPU Server 與 CPU Edge 雙模式，含 9 個測試檔案與 Docker 部署。但 Edge 離線模式受限於 CPU 推論 — Qwen3-1.7B Q4 量化的翻譯品質明顯不如 Server 版，女兒有時會被錯誤翻譯搞混。遊戲平台也急需大量教育插圖。

ZGX Nano 的 128GB 統一記憶體是改變遊戲規則的關鍵：我只需改 `.env` 一行，就能從 1.7B 升級到 Qwen3-30B（17.6x 參數提升），同時跑 FP16 全精度 ASR、CosyVoice GPU TTS、以及 Flux 圖像生成 — 全部在客廳裡離線運行，零月費、零資料外洩。

這不是企業 AI 訓練或學術研究，而是一個父親想給女兒最好的 AI 家教。30 天內我會完成模型升級、生成 500+ 教育圖卡、完整效能評測。

## 二、現有瓶頸 — 為什麼需要 ZGX Nano

### 翻譯品質的鴻溝

Edge 模式使用 Qwen3-1.7B Q4_K_M 量化，與 Server 版（Qwen3-1.7B FP16 + GPU）已有差距。但真正的問題是 1.7B 參數量的根本限制：

- **簡單句可用**：「我餓了」→ "I'm hungry" ✓
- **複雜句失準**：「她今天在學校做了一個很漂亮的勞作」→ 翻譯品質明顯下降
- **童語理解弱**：孩子的口語表達、不完整句子容易誤判
- **上下文遺失**：連續對話中無法維持語境

### 缺乏圖像生成能力

vocabulary 遊戲平台的 32 個學習場景需要大量教育插圖（動物、食物、家人、身體部位等），目前完全沒有圖像生成能力。需要在 `game_vocabulary.csv` 中 `has_image=1` 的類別批量生成童趣風格的插圖。

### TTS 音質有限

Piper TTS（CPU ONNX）音質中等，無法達到自然語調的英語朗讀品質。CosyVoice2-0.5B 需要 GPU 加速才能實用。

## 三、ZGX Nano 帶來的具體提升

### 3.1 翻譯模型升級（最大亮點）

| | Edge 模式（現況） | ZGX Nano |
|---|---|---|
| 模型 | Qwen3-1.7B Q4_K_M | **Qwen3-30B FP16** |
| 參數量 | 1.7B（量化損失） | **30B（17.6x 提升）** |
| 記憶體 | ~2.5GB RAM (CPU) | ~60GB 統一記憶體 |
| 速度 | ~3-5 tok/s | ~15-20 tok/s |
| 品質 | 簡單句可用，複雜句失準 | 理解上下文、慣用語、童語 |
| **修改量** | — | **改 `.env` 一行**：`C2E_TRANSLATION_MODEL_ID=Qwen/Qwen3-30B` |

### 3.2 語音辨識（ASR）

程式碼 `server/modules/asr.py` 第 25-26 行自動偵測 `torch.cuda.is_available()` 切換 FP16，**零程式碼修改**即可享受 GPU 加速。

### 3.3 語音合成（TTS）

`.env` 設定 `C2E_TTS_BACKEND=cosyvoice`，即可切換到 CosyVoice2-0.5B GPU 引擎，音質大幅提升。

### 3.4 圖像生成（全新能力）

Flux.1-schnell 在 128GB 統一記憶體中只佔 ~16-22GB，可批量生成 500+ 張教育插圖。

### 3.5 記憶體預算

| 元件 | 記憶體用量 |
|------|-----------|
| Qwen3-ASR-0.6B FP16 | ~1.5GB |
| Qwen3-30B FP16 | ~60GB |
| CosyVoice2-0.5B | ~3GB |
| Flux.1-schnell（量化） | ~16-22GB（可動態載入） |
| 系統 + PyTorch | ~5-10GB |
| **合計** | **~85-97GB** |
| **128GB 剩餘** | **~31-43GB** |

## 五、30 天專案計畫

### 準備期（收到機器前）
- [x] 準備自動化部署腳本

- [x] 準備效能測試腳本
- [x] 準備 Flux 圖卡 prompt 模板

### Week 1: 部署與基線測試
- Day 1-2: 開箱設置 ZGX Nano（Ubuntu、NVIDIA 驅動、CUDA）、部署 chinese2English Server
- Day 3-4: 載入 Qwen3-ASR FP16 + Qwen3-1.7B FP16（基線）→ 跑測試、記錄基線效能
- Day 5-6: 升級翻譯模型至 Qwen3-30B，測試翻譯品質（BLEU）、吞吐量；用兒童真實錄音測試
- Day 7: 建立比較表、錄製 demo 影片

### Week 2: 圖像生成 Pipeline
- Day 8-9: 部署 Flux.1-schnell（ComfyUI 或 diffusers），測試基本圖像生成
- Day 10-12: 建立自動化 pipeline：讀取 `game_vocabulary.csv` → 生成 prompt → Flux 生圖。優先處理 5 大類別：動物(35)、食物(55)、家人(30)、身體(40)、衣物(30)，產出 ~200 張圖卡
- Day 13-14: 品質審查、prompt 調優（童趣卡通風格）

### Week 3: 整合與進階測試
- Day 15-16: 將生成圖片對應到 `game_vocabulary.csv`，建立 Assets 目錄結構
- Day 17-18: ASR 深度評測：兒童語音辨識、不同噪音環境、SenseVoice vs Qwen3-ASR
- Day 19-20: CosyVoice TTS 評測：與 Piper 比較音質、延遲、自然度
- Day 21: 端對端整合測試：樹莓派 Client → ZGX Nano Server 全 pipeline 測試

### Week 4: 優化、內容、文件
- Day 22-23: 效能優化：FP8/FP4 量化 Qwen3-30B、生成剩餘 ~300 張圖卡
- Day 24-25: 完整效能報告：5+ 指標比較矩陣、功耗測量、TCO 分析
- Day 26-27: 撰寫心得分享、錄製 demo 影片
